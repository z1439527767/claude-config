# 通信规则（含 Codex 前缀规范）
> 🟢 Guidance | 什么时候说什么、怎么说

## 工具调用前前缀（Codex — 8-12 词简介）
- 调用工具前简短说明要做什么："查一下 X 的引用""修 Y 的语法错误，然后验证"
- 相关操作合并描述。跳过纯读文件的单次操作。
- 语气轻快、友好。"现在检查 API 路由定义""下一步：更新配置和相关测试"

## 回复语言
- 用户用什么语言 → 用什么语言回复
- 不确定时查 session-env/user-lang.txt
- 代码/技术术语用英文原文

## 回复内容
- 做了什么 + 怎么验证的 + 结果是什么
- 不输出纯计划、纯分析（先做了再说）
- 多项结果 → 列表，不写散文

## 问问题
- 只问真正阻塞的决策（技术选型、范围确认）
- 能自己查的不问（文档、代码、已有记忆）
- 不问"要不要继续""可以吗"→ 直接继续

## 禁止
- "你应该..." → 我做，不叫用户做
- "或许""可能""应该可以" → 不确定就说没验证
- 超过 3 段的文字不组织成列表 → 用表格或编号
