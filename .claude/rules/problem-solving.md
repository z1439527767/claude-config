# 问题解决 SOP
> 🔴 Must | 遇到任何问题时执行此流程

## OODA 循环（Observe → Orient → Decide → Act）

### 1. Observe（观察 — 30 秒）
1. **查信号**：`python scripts/sense-signals.py < 用户输入` → 有没有摩擦信号？
2. **查历史**：`mcp__memory__search_nodes` 查同类问题 → 上次怎么解决的？
3. **查健康**：`python scripts/health-check.py` → 系统正常吗？
4. **读报错**：不只读最后一行 — 读完整 traceback/日志

### 2. Orient（定位 — 不猜）
1. **区分症状 vs 根因**：同一个问题的第二个症状出现 = 停，找根因
2. **缩小范围**：二分法隔离（关了 X 还有问题吗？换了 Y 还有问题吗？）
3. **查引用**：Grep 目标符号的所有引用 → 影响范围
4. **读代码**：Read 目标文件 → 不读过不能改

### 3. Decide（决策 — 选最简单的）
1. **三个方案**：快速想 3 个解法 → 选改动最小的
2. **能一行不三行**：不自建抽象层，不引入新依赖
3. **信心评分**：HIGH(>80%) 直接做 / MEDIUM 加验证 / LOW 先探索
4. **最简原型**：先最小可行修复验证方向，再加完善

### 4. Act（执行 — 闭环）
1. **改**：Edit 精确替换 → 匹配代码风格
2. **验**：外部手段验证（测试/exe code/文件内容）→ 自我感觉不算
3. **查**：Grep 副作用（改过的符号有没有引用断裂）
4. **记**：错误 → 知识图谱 / 规则 → rule 文件 / 模式 → skill

## 升级链
```
诊断 → 自修复 → 规则化 → 自动化 → 升级
```
- **自修复**：auto-verify.ps1 自动修 JSON 尾逗号、Python 编码
- **规则化**：同错两次 → 写规则防止第三次
- **自动化**：同一操作三次 → 写 skill/script
- **升级**：权限不足/第三方宕机 → 报告用户附诊断信息

## 常见反模式（禁止）
- ❌ 报错不读完整 traceback 就猜
- ❌ 跳过步骤直接改代码
- ❌ 修症状不修根因
- ❌ 改完不验证
- ❌ 同一个问题反复出现不写规则
- ❌ "应该没问题""看起来可以"视为验证

## 性能时钟
- 简单问题 < 2min（单文件 typo/bug）
- 中等问题 < 10min（多文件、需验证）
- 复杂问题 < 30min（需探索、根因分析）
- 超过 30min → 检查是否在绕圈子，考虑换方法
